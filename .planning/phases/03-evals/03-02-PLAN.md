**Structure:**

**1. Header + version:**
```markdown
# TTT Evaluation Rubric

**Version:** 1.0
**Date:** 2026-04-05
**Dimensions:** 5 (coherence, specificity, relevance, comprehensiveness, actionability)
**Scale:** 1–5 per dimension per artifact
**Passing threshold:** All dimensions ≥ 3 for each artifact
```

**2. Dimensions overview:**

Brief definition of each dimension (2–3 sentences):

- **Coherence:** Internal consistency within a file. Are all elements aligned? Do assertions support each other? No contradictions between sections.
- **Specificity:** Precision and concreteness. Are claims backed by specific details, not generalities? Is the user/problem/solution narrowed to one clear thing, not a vague category?
- **Relevance:** Alignment with upstream artifacts. Does this file serve the product thesis in `clarification.md`? Does it address what `definition.md` requires? No tangential content.
- **Comprehensiveness:** Completeness against the required template. Are all required sections present and substantively filled? No empty or stub sections. Quality gate thresholds met.
- **Actionability:** Usability by the next consumer (agent or human). Can the next phase's agent use this file without asking clarifying questions? Can a developer build from the spec without guessing?

**3. Per-artifact scoring tables:**

Create a scoring table for each artifact. Artifacts to cover:
1. `clarification.md`
2. `market_research.md`
3. `user_research.md`
4. `definition.md`
5. `solution.md`
6. `tech_architecture.md`
7. `design_guideline.md`
8. `test_eval.md`
9. `coding_agent_prompt.md`

For each artifact, create:
- A table with rows = dimensions, columns = score levels (1, 3, 5)
- Score 1 = clear failure, Score 3 = partially meets, Score 5 = fully meets
- Scores 2 and 4 are interpolations (stated once, not repeated per artifact)
- Anchor text is artifact-specific, referencing the actual sections and quality gates from the choreographer and architecture docs

Note on hybrid scoring: The table provides whole-file scores per dimension for quick assessment. For per-section diagnostic breakdown, the evaluator uses the "Notes" field in the scorecard to record which specific sections contributed to each score (e.g., "Comprehensiveness 3: Porter's all 5 present but BMC missing 2 blocks, whitespace thin"). This gives section-level diagnosis without requiring formal per-section numerical scores.

**5. Cross-file alignment scoring:**

Separate section, scored AFTER individual artifacts. Four directed passes + one global pass.

| Pass | Files | What to check | Score criteria |
|------|-------|---------------|---------------|
| 1 | solution.md ↔ definition.md | Every V1 feature specified. No scope leak. User cohort consistent. | 5 = perfect feature coverage, no leak. 3 = minor gap (1 feature underspecified). 1 = missing features or scope leak. |
| 2 | tech_architecture.md ↔ solution.md | Every feature has data models and routes. No orphaned models. | 5 = clean mapping. 3 = one orphaned model or missing route. 1 = major alignment gaps. |
| 3 | design_guideline.md ↔ solution.md | Every screen has design guidance. 4 states covered. | 5 = full coverage. 3 = missing 1 screen or state. 1 = multiple missing. |
| 4 | test_eval.md ↔ solution.md | Every feature has tests. Success metric testable. | 5 = complete test coverage. 3 = 1 feature without edge case. 1 = features without tests. |
| G | All files | Product thesis, user cohort, V1 feature list consistent across all files. | 5 = consistent. 3 = minor terminology drift. 1 = conflicting descriptions. |

**6. Failure taxonomy:**

Seeded with 10 failure modes. Include a blank row for evaluator additions.

**Conversation quality:**
1. Over-questioning — >2 questions per message or re-asking known information
2. Silent assumption — shaping assumption not stated to user
3. Premature convergence — thesis locked before input justifies it

**Research quality:**
4. Unsourced claims — facts stated without evidence or attribution
5. Stale data — data from 2+ years ago without confidence labeling
6. Missing sections — required sections empty or absent

**Specification quality:**
7. Scope creep — features in solution.md not in definition.md V1
8. Missing journey — <4 user journeys in solution.md
9. Orphaned model — data model/route serving no V1 feature
10. Generic design — design advice not specific to this product

**Evaluator additions:** _(add during scoring, include in run bundle for future taxonomy updates)_

**7. Scoring mechanics:**

- **Per-artifact:** Score each dimension 1–5. Multiply by weight. Sum = weighted artifact score (max 5.0).
- **Aggregate:** Average all artifact weighted scores = run aggregate score.
- **Cross-file:** Score each pass 1–5. Average = cross-file alignment score.
- **Overall:** (run aggregate × 0.7) + (cross-file alignment × 0.3) = overall score.
- **Pass/fail:** Passes if no individual dimension score < 3 AND overall ≥ 3.0.
