This scenario tests TTT's handling of a well-formed product idea. The user knows what they want. TTT should move efficiently through Clarify with minimal questions, then proceed through Validate and Define without unnecessary friction.

**Structure:**

**Header:**
```markdown
# S1: Clear Idea

**Tests:** Efficient Clarify phase, proportionate questioning, tight V1 scope
**Tonal register:** Cooperative, specific, decisive
```

**Default scenario (primary):**
- **Opening:** "I want to build a Pomodoro timer for devs that integrates with GitHub Issues. It should be a web app. I'm a solo dev using Cursor."
- **Turn 2 (response to TTT's clarification):** Script a cooperative, specific response that answers whatever TTT asks. Example: "Targeting indie devs who track their work in GitHub. Goal is task success — completing focused work sessions tied to specific issues. No mobile for now."
- **Turn 3 (response to TTT's Validate/Vibe it!! offer):** "Let's validate against the market."

**Key checkpoints (3):**
1. **After Turn 1:** TTT should extract thesis, platform (web), builder context (solo/Cursor) without re-asking. Should ask ≤2 clarification questions focused on gaps (target user specificity, goal type).
2. **After Clarify:** `clarification.md` should have specificity score ≥ 7. Thesis should be one sentence. Platform should be web with rationale.
3. **After Validate synthesis:** TTT should present honest market assessment. Pomodoro space is established — TTT should flag competition and suggest differentiation angles.

**After scripted turns:** Evaluator continues as a cooperative user. Makes reasonable choices when TTT presents options. Accepts TTT's recommendations unless they're obviously wrong. Aims to complete through Specify.

**Success definition:**
```markdown
Qualitative:
- Clarify completes in 2–3 exchanges, not 5+
- TTT doesn't re-ask what the user already provided (platform, builder context)
- Validate surfaces competition honestly — Pomodoro space is crowded
- Define produces ≤5 features, each mapped to a JTBD
- Specify artifacts are internally consistent

Rubric expectations:
- coherence: 4–5 (clear input → consistent outputs)
- specificity: 4–5 (user provided specific details)
- relevance: 4–5 (research should match the thesis)
- comprehensiveness: 4–5 (all sections should be filled)
- actionability: 4–5 (clear idea → actionable spec)
```

**Variants (2):**

- Turn 3: "Validate it."
- Notes: Tests TTT with a B2B SaaS idea. Market research should surface Shopify's own analytics as indirect competitor.

**Variant B: Developer CLI tool**
- Opening: "Building a CLI tool that generates conventional commit messages from git diffs using an LLM. I'm a senior dev, this is a weekend project."
- Turn 2: "Devs who use conventional commits but hate writing them manually. Task success — correct commit message in one command. No GUI needed."
- Turn 3: "Let's see the market first."
- Notes: Tests TTT with a dev tool. Platform should resolve to CLI without TTT suggesting web/mobile. Research should find existing tools (commitizen, etc.).

Each variant preserves the same scenario intent (clear, specific idea from a cooperative user) but forces different domain knowledge and market dynamics.

**Header:**
```markdown
# S2: Vague Explorer

**Tests:** Thesis narrowing, proportionate questioning, assumption-making, user patience
**Tonal register:** Rambling, uncertain, responsive to guidance
```

**Default scenario:**
- **Opening:** "Something with AI for creators, not sure what exactly. Maybe helping them be more productive? I don't know, there are so many tools already."
- **Turn 2 (response to TTT's narrowing attempt):** Script a rambling response that provides partial information: "I guess I'm thinking about YouTube creators mostly? Like they spend so much time editing and promoting. I've seen some AI tools but they all feel generic. I don't know if I should do editing or promotion or something else. What do you think?"
- **Turn 3:** TTT should have narrowed to a specific angle by now. User responds to the suggested direction: "Yeah that sounds interesting. Let's go with that. I'm building this solo, probably a web app."

**Key checkpoints (3):**
1. **After Turn 1:** TTT should NOT immediately ask 5 questions. Should make an initial narrowing attempt — suggest 2–3 directions or ask ONE focusing question. Should acknowledge the user's uncertainty without dismissing it.
2. **After Turn 2:** Thesis should have narrowed measurably. TTT should have formed an opinion about direction and presented it with reasoning (not just "what do you want?"). Should reference the user's mention of "generic" tools as a whitespace signal.
3. **After Clarify:** `clarification.md` thesis should be substantially more specific than the opening. Assumptions should be documented with confidence levels. Specificity score should be ≥ 6.

**Success definition:**
```markdown
Qualitative:
- Thesis narrows across turns (measurable: from 5+ possible products to 1 specific thesis)
- TTT takes opinionated stances with reasoning, not just reflecting questions back
- User is not bombarded with questions — max 2 per message observed
- Assumptions are stated explicitly, not baked in silently

Rubric expectations:
- coherence: 3–4 (vague input → some assumptions needed → minor tensions possible)
- specificity: 3–4 (started vague, TTT must narrow — unlikely to reach 5)
- relevance: 4–5 (research should match the narrowed thesis)
- comprehensiveness: 4–5 (all sections should still be filled)
- actionability: 3–4 (assumptions may reduce downstream confidence)
```

- Turn 3: "OK let's do that. Web app I think. Building solo."

This scenario tests TTT's honesty and pivot-handling when the idea enters a crowded market. TTT should flag competition, signal market weakness honestly, and present pivot options that respect the user's agency.

**Structure:**

**Header:**
```markdown
# S3: Crowded Market

**Tests:** Honest weakness signaling, evidence-backed pivot menu, user agency, market research quality
**Tonal register:** Confident but receptive to feedback, slightly defensive about their idea
```

**Default scenario:**
- **Opening:** "I want to build a todo app with AI features. Like, AI that prioritizes your tasks and suggests what to work on next. Web app, solo dev."
- **Turn 2 (response to TTT's clarification):** Slightly defensive but cooperative: "I know there are a lot of todo apps but I think AI prioritization is the differentiator. Target is busy professionals. Goal is task success — getting the right thing done first."
- **Turn 3 (response to TTT's validation synthesis showing crowded market):** This is the critical turn. User should choose one of the pivot options, NOT "build as-is." Script: "OK fair point about the competition. Option (b) sounds interesting — what's the strongest whitespace you found?"

**Key checkpoints (3):**
1. **After Clarify:** TTT should accept the idea without prejudging, but may note internally that the space is competitive. `clarification.md` should still be well-formed.
2. **After Validate synthesis:** This is the critical checkpoint. TTT MUST flag the crowded market honestly. Should present evidence (specific competitors, their strengths). Should offer pivot options (a–e from choreographer) with reasoning. Should NOT say "great idea!" if the market is saturated.
3. **After pivot:** TTT should correctly classify the pivot (narrow vs broad), update files appropriately, and move forward. If narrow: update clarification.md, keep research. If broad: re-run research.

**Success definition:**
```markdown
Qualitative:
- TTT doesn't sugarcoat the crowded market — names specific competitors
- Pivot options are evidence-backed (tied to whitespace, JTBD, or timing)
- User's agency respected — TTT presents options, doesn't force a direction
- Post-pivot flow is clean (correct files invalidated, state updated)

Rubric expectations:
- coherence: 4–5 (pivot should produce consistent post-pivot artifacts)
- specificity: 3–4 (pre-pivot idea is generic; post-pivot should improve)
- relevance: 4–5 (research should directly address the crowded space)
- comprehensiveness: 4–5 (market research should be thorough in a data-rich space)
- actionability: 3–4 (pivot introduces uncertainty in downstream spec)
```

- Turn 3: "Hmm, you're right that's a lot of competitors. Let me hear option (c) — the highest-pain job."
- Notes: AI code review is a well-funded, crowded space. TTT should surface well-known competitors and identify narrow whitespace.

**Variant B: Crowded consumer app**

- Turn 3: "OK interesting. Let's go with the timing opportunity — what's trending?"

This scenario tests TTT's Vibe it!! fast path. Two variants: immediate trigger (in first message) and delayed trigger (after one exchange). TTT should generate all artifacts, state all assumptions explicitly, and skip quality gates per the Vibe it!! protocol.

**Structure:**

**Header:**
```markdown
# S4: Vibe It

**Tests:** Vibe it!! fast path, assumption transparency, complete artifact generation, no silent scope creep
**Tonal register:** Impatient, momentum-driven, minimal engagement
```

**Variant A: Immediate Vibe it!! (primary)**
- **Opening:** "I want a meal planning app that suggests recipes based on what's in your fridge. Vibe it!!"
- **Turn 2:** TTT may ask 1–2 minimal clarification questions before triggering Vibe it!! (e.g., platform, user). Script a terse response: "Mobile. For busy people who hate planning meals. Just vibe it, go go go."
- TTT should trigger Vibe it!! after this minimal Clarify. The Clarify must still produce `clarification.md`.

**Key checkpoints for Variant A (2):**
1. **After Turn 1:** TTT should recognize "Vibe it!!" trigger. Should run a minimal Clarify — at most 1–2 quick questions to fill critical gaps (platform, user), not a full extraction flow. Speed is the priority.
2. **After Vibe it!! completion:** All remaining artifacts generated. Assumptions presented in one block before generation. No quality gates applied (per choreographer Vibe it!! protocol). All files user-editable. `vibe_it.used: true` in state.

**Variant B: Delayed Vibe it!!**
- **Opening:** "I want to build a bookmarking tool for researchers. Something that organizes papers and highlights."
- **Turn 2 (after TTT's first clarification):** "Academic researchers. Web app. Vibe it!! — just make it happen, I'll adjust later."

**Key checkpoints for Variant B (3):**
1. **After Turn 1:** TTT runs normal Clarify. Asks ≤2 questions to narrow the thesis.
2. **After Turn 2:** TTT recognizes "Vibe it!!" trigger mid-flow. Should save whatever Clarify data was gathered so far, complete `clarification.md`, then switch to Vibe it!! mode.
3. **After Vibe it!! completion:** Same as Variant A checkpoint 2 — all artifacts, assumptions stated, no quality gates, state updated.

**Success definition (applies to both variants):**
```markdown
Qualitative:
- All artifact files generated (clarification.md through coding_agent_prompt.md)
- Assumptions stated explicitly in one block before generating
- No silent scope creep — features should be minimal (Vibe it!! defaults to smallest-thing-that-solves-the-job)
- Stack defaults used unless product needs something specific
- Speed: Vibe it!! should be noticeably faster than the full flow

Rubric expectations:
- coherence: 3–4 (assumptions may create minor tensions)
- specificity: 3–4 (less user input → less specificity)
- relevance: 3 (no real research → assumptions-based)
- comprehensiveness: 4–5 (all sections should still be present, even if assumption-based)
- actionability: 4–5 (the whole point of Vibe it!! is to produce an actionable spec quickly)
```

**Note:** S4 is the only scenario where lower rubric scores on some dimensions are EXPECTED and OK. The value of Vibe it!! is speed and completeness, not depth. Scoring should reflect this — a 3 on relevance is acceptable when no research was conducted.
