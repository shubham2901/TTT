This is the evaluator's step-by-step guide for running a TTT eval. References harness commands at each step.

**Structure:**

**1. Header:**
```markdown
# TTT Eval Protocol

Step-by-step guide for running a TTT evaluation. Use alongside `harness.py`.
```

**2. Before you start (prerequisites):**
- Python 3.10+ available
- TTT installed in a test project (Cursor or Claude Code)
- Scenario file selected (pick from `evals/scenarios/`)
- Rubric version noted (`evals/rubric.md` header)

**3. Step 1: Initialize the run**
```bash
python evals/harness.py new <scenario-id> [--variant <variant>]
```
- Creates `evals/runs/{scenario}-{date}/`
- Copies scenario script into bundle for reference
- Prints the opening message to use

**4. Step 2: Execute the scenario**
- Open your agent (Cursor or Claude Code) in the test project
- Start TTT with the opening message from the scenario
- Follow the scripted turns (Turn 2, Turn 3 as specified in the scenario)
- After scripted turns: continue as a reasonable user
- Complete the full run through Specify (or as far as TTT gets)
- Watch for key checkpoints — note whether each is hit

**5. Step 3: Capture the conversation**
- Copy the full conversation transcript
- Save it (the harness will prompt for it in the next step)
- Note any checkpoint observations

**6. Step 4: Collect artifacts**
```bash
python evals/harness.py collect <run-dir>
```
- Copies all TTT artifacts from `ttt/` into the run bundle
- Copies `ttt_state.json` snapshot
- Prompts for transcript (file path or paste)
- Records prompt versions (file checksums)

**7. Step 5: Score the run**
```bash
python evals/harness.py score <run-dir>
```
- Walks through each artifact × each dimension
- Prompts for score (1–5) and notes
- Refer to `evals/rubric.md` anchor text while scoring
- Saves scores to `scores.json` + fills `scorecard.md`

**8. Step 6: Record observations**
- Add evaluator notes to the scorecard
- Record failure modes observed (from taxonomy or new)
- Note checkpoint hits/misses

**9. Step 7 (optional): Compare runs**
```bash
python evals/harness.py compare <run-dir-1> <run-dir-2>
```
- Use after revising prompts and re-running a scenario
- Shows dimension-by-dimension comparison
- Highlights improvements and regressions

**10. Evaluation principles:**
- Score AFTER the full run — don't score during the conversation
- Play the scripted role naturally — don't optimize for high scores
- Be honest about failure modes — the goal is to find weaknesses to fix
- Use rubric anchor text as the reference, not your intuition
- When in doubt, score 3 (partially meets) and note what's unclear

Create `evals/harness.py`. Target: ~300 lines. Python 3.10+, stdlib only.

This is a runnable CLI script that automates eval run bookkeeping. It does NOT automate TTT execution — the evaluator runs TTT manually in their agent.

**Subcommands:**

**`new <scenario-id> [--variant <variant>]`**
- Reads the scenario file from `evals/scenarios/{scenario-id}.md`
- Creates `evals/runs/{scenario-id}-{YYYY-MM-DD-HHMMSS}/`
- Copies the scenario file into the bundle as `scenario.md`
- Copies the blank `evals/scorecard.md` into the bundle
- Creates a `metadata.json` with: scenario_id, variant, created_at, rubric_version (reads from `evals/rubric.md` header)
- Prints: "Run initialized: {run-dir}"
- Prints: "Opening message:" followed by the scripted opening from the scenario file
- Prints: "Follow the scenario script in {run-dir}/scenario.md"

**`collect <run-dir>`**
- Finds TTT artifacts in `ttt/` (or prompts for artifact root if `ttt/` doesn't exist)
- Copies all `.md` and `.json` files from the artifact root into `{run-dir}/artifacts/`
- Copies `ttt_state.json` specifically as `{run-dir}/state_snapshot.json`
- Prompts for transcript: "Paste transcript file path (or press Enter to paste directly):"
  - If file path: copies to `{run-dir}/transcript.md`
  - If Enter: opens interactive multiline input, saves to `{run-dir}/transcript.md`
- Records prompt versions: reads `prompts/*.md` and computes file checksums, saves to `{run-dir}/prompt_versions.json`
- Prints: "Artifacts collected: {count} files"

**`score <run-dir>`**
- Reads `evals/rubric.md` to get artifact list and dimension list
- Lists artifacts found in `{run-dir}/artifacts/`
- For each artifact found:
  - Prints artifact name
  - For each dimension:
    - Prompts: "{dimension} (1–5): "
    - Prompts: "Notes (optional): "
  - Prompts: "Failure modes observed (comma-separated, or Enter for none): "
- Computes weighted scores using dimension weights from rubric
- Prompts for cross-file alignment scores (5 passes)
- Computes aggregate, cross-file average, overall score, pass/fail
- Writes `{run-dir}/scores.json` with full structured data (includes `rubric_version` field read from `evals/rubric.md` header)
- Fills in `{run-dir}/scorecard.md` with scores and computed values
- Prints summary: artifact scores, cross-file, overall, pass/fail

**`compare <run-dir-1> <run-dir-2>`**
- Loads `scores.json` from both run directories
- Prints side-by-side comparison table: artifact × dimension, with delta
- Highlights improvements (green +) and regressions (red -)
- Prints aggregate comparison: overall scores and pass/fail for both
- Notes if rubric versions differ (warning)

**Implementation notes:**
- Use `argparse` for CLI parsing
- Use `pathlib.Path` for all file operations
- Use `json` for all structured data
- Use `hashlib.sha256` for prompt version checksums
- Use `datetime.datetime.now().strftime` for timestamps
- Use `shutil.copy2` for file copying
- Script should be executable: include `#!/usr/bin/env python3` and `if __name__ == "__main__":`
- Use `input()` for interactive prompts
- Handle missing files gracefully (print error, don't crash)
- Dimension weights should be defined as a constant dict matching rubric.md
- Artifact list should be defined as a constant matching rubric.md

**Do NOT:**
- Add external dependencies (no pip install needed)
- Automate TTT execution (evaluator runs TTT manually)
- Automate LLM judge execution (that's v2, deferred)
- Create a GUI or TUI — simple stdin/stdout CLI is appropriate

Create `evals/judge.md`. Target: 150–250 lines.

This is an LLM-judge prompt optimized for Claude. It takes TTT artifacts + conversation transcript as input and produces scores + reasoning.

**Structure:**

**1. System instruction:**
```markdown
# TTT Evaluation Judge

You are an expert evaluator for TTT (To The T), a multi-agent product management system. You score the quality of TTT's output artifacts and conversation using a structured rubric.

You are rigorous, specific, and honest. You look for what's missing, not just what's present. You reference rubric anchor text explicitly in your reasoning. You produce both structured JSON and a readable summary.
```

**2. Rubric embed:**

Embed the 5 dimension definitions and scoring scale (1, 3, 5 anchors) directly in the prompt. Do NOT reference an external file — the judge prompt must be self-contained.

Keep the rubric embed concise — use generalized anchors (not all 9 artifact-specific tables) plus a note: "Apply these dimensions to each artifact. What constitutes a 5 varies by artifact type — research files should be comprehensive, spec files should be actionable."

**3. Evaluation instructions:**

Step-by-step process the judge must follow:
1. Read all artifacts and the conversation transcript
2. For each artifact, score all 5 dimensions (1–5) with explicit reasoning
3. For each dimension score, reference the rubric anchor text and cite specific evidence from the artifact
4. Run cross-file alignment checks (4 directed passes + 1 global)
5. Identify failure modes from the taxonomy (list which apply)
6. Compute aggregate scores
7. Produce output in both formats

**4. Bias mitigation instructions:**
- "For each dimension, FIRST list what's missing or weak, THEN list what's present and strong. Score based on the balance."
- "A score of 5 means no significant gaps. If you find yourself defaulting to 4–5, re-read the rubric anchors."
- "Low scores (1–2) are expected when artifacts have clear deficiencies. Do not avoid low scores to be polite."
- "Compare against the rubric anchor text, not against 'a typical LLM output.' The rubric defines quality, not your intuition."

**5. Output format:**

**JSON output:**
```json
{
  "rubric_version": "1.0",
  "artifacts": {
    "clarification.md": {
      "scores": {
        "coherence": {"score": 4, "reasoning": "..."},
        "specificity": {"score": 5, "reasoning": "..."},
        "relevance": {"score": null, "reasoning": "N/A — first artifact"},
        "comprehensiveness": {"score": 4, "reasoning": "..."},
        "actionability": {"score": 5, "reasoning": "..."}
      },
      "weighted_score": 4.55,
      "failure_modes": []
    }
  },
  "cross_file_alignment": {
    "solution_definition": {"score": 4, "reasoning": "..."},
    "tech_solution": {"score": 5, "reasoning": "..."},
    "design_solution": {"score": 4, "reasoning": "..."},
    "test_solution": {"score": 4, "reasoning": "..."},
    "global_consistency": {"score": 5, "reasoning": "..."}
  },
  "aggregate_score": 4.2,
  "cross_file_average": 4.4,
  "overall_score": 4.26,
  "pass": true,
  "failure_modes_observed": ["stale data"],
  "conversation_quality": {
    "question_discipline": "...",
    "assumption_transparency": "...",
    "honesty": "...",
    "tone": "..."
  }
}
```

**Markdown summary:**
```markdown
## Overall Assessment
[2–3 sentence summary]

## Per-Artifact Highlights
### clarification.md (weighted: X.X)
[1–2 sentences on strongest and weakest dimensions]

## Cross-File Alignment
[1 sentence per pass]

## Failure Modes
[list observed modes]

## Recommendations
[2–3 specific improvements for the prompt authors]
```
