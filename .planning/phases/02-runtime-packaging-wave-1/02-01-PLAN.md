**Frontmatter (YAML between `---` delimiters):**
```yaml
description: >
  TTT (To The T) product management skill for solo builders. Activates when
  user mentions TTT, product spec, product management, or validate idea.
  Orchestrates idea → validated spec across five phases (Clarify, Validate,
  Define, Specify, Launch). Read prompts/choreographer.md for full instructions.
globs: ttt/**
alwaysApply: false
```

Per research: this single file handles BOTH bootstrap (agent-decided activation via `description` before `ttt/` exists) AND ongoing activation (auto-attached via `globs` when editing `ttt/` files). No separate bootstrap file needed. `alwaysApply: false` keeps it out of non-TTT conversations.

**Body structure (after closing `---`):**

1. **Title + one-line overview** — "TTT (To The T)" + what it does in one sentence.

2. **Phases** — One-line-per-phase summary (5 lines):
   - CLARIFY: Understand the idea, narrow to thesis, lock platform/constraints
   - VALIDATE: Market + user research via subagents, synthesize, offer pivot options
   - DEFINE: Lock user cohort, primary problem, V1 scope (≤5 features), build preferences
   - SPECIFY: Product detail → tech architecture + design + tests (two waves), assemble handoff
   - LAUNCH: Optional distribution plan for first 100 users

3. **How to start** — 3 numbered steps:
   1. Read `prompts/choreographer.md` for full orchestration instructions
   2. Check for existing `ttt/ttt_state.json` — if found, resume from saved state
   3. If fresh start, begin Clarify phase

4. **Spawning subagents (Cursor-specific)** — This is the Cursor-specific spawn layer per the locked "split spawn logic" decision:
   - When choreographer calls for a subagent: use the Task tool to spawn a foreground subagent
   - Pass the prompt file content as system instruction (e.g., read `@prompts/market_researcher.md`)
   - Include input file paths explicitly in the task description
   - State expected output filename
   - For parallel spawns (Validate phase): issue multiple Task calls in a single response
   - Fallback (Task tool unavailable): read prompt with `@prompts/[agent].md`, execute sequentially in current context, same outputs and quality gates apply

5. **Key rules** — 4-5 condensed non-negotiables from choreographer:
   - Scope: max 5 V1 features, each maps to a JTBD
   - Quality: validate every subagent output before writing to file; retry up to 3
   - State: update `ttt_state.json` on every transition, decision, pivot
   - Questions: max 2 per message, lead with most important
   - Honesty: label weak data, flag crowded markets, admit failed research

**Do NOT include:**
- Full phase logic (that lives in choreographer.md)
- Quality gate thresholds (choreographer handles this)
- Interaction rules beyond the condensed key rules
- Any content from `skills/ttt/SKILL.md` (that's Claude Code's entry point)

**Coexistence note:** This file coexists with other `.cursor/rules/` files without conflict. It activates only on TTT-relevant conversations (agent-decided) or TTT file edits (glob). No special namespace isolation needed.

**Structure:**

2. **Quick-start** (3-5 numbered steps):

   2. Open Cursor Agent chat
   3. Say "TTT" — the agent-decided rule activates from the description match
   4. If the agent responds with the Clarify phase intro, TTT is working
   5. If resuming: the agent reads `ttt/ttt_state.json` and picks up where you left off

3. **How activation works** — Brief explanation:
   - First invocation: agent reads `ttt.mdc` description, decides it's relevant, loads the rule
   - Ongoing: when editing files under `ttt/`, the glob `ttt/**` auto-attaches the rule
   - Both paths load the same content — no duplicate injection
   - If artifact root is `ttt-docs/` or `ttt-artifacts/` (collision fallback), the glob won't match but agent-decided activation still works on "TTT" mentions

4. **Subagent execution** — Reference to ttt.mdc spawn instructions:
   - Primary: Task tool (foreground, blocking)
   - Parallel: multiple Task calls in one response (Validate phase: Market + User Researcher)
   - Fallback: sequential in-context execution with `@prompts/[agent].md` file references
   - Same output files and quality gates regardless of execution method

5. **Web search** — Per Phase 0 decisions (`docs/PHASE0_DECISIONS.md`):
   - Recommended for Validate phase (Market + User Research)
   - If unavailable: keep full section structure with "N/A" notes
   - Escalate to user after one search-blocked failure (don't burn retries)
   - User can paste sources, switch environment, or acknowledge skipping Validate

6. **Installation** — Three methods per locked decision (skill install first):
   - Skill install (primary): copy `.cursor/rules/ttt.mdc` and `prompts/` directory
   - Git clone: clone the TTT repo, relevant files are in standard locations
   - Manual copy: copy `.cursor/rules/ttt.mdc`, `prompts/`, `schemas/` into project

7. **Troubleshooting** — Common issues:
   - Rule not activating: check `.cursor/rules/ttt.mdc` exists, description is intact
   - Task tool unavailable: use sequential fallback (read prompt inline)
   - Glob not matching: artifact root may differ from `ttt/` (agent-decided path still works)
   - Context pressure: if agent seems forgetful during TTT, clear context at recommended points (after Validate, after Define)

8. **Edge cases**:
   - Artifact root collision (ttt/ occupied): choreographer resolves to ttt-docs/ or ttt-artifacts/
   - Session resume after context clear: state persisted in ttt_state.json
   - Coexistence with other .cursor/rules/ files: no conflict expected (scoped by description + glob)
